1. Bản chất thiết kế: Hợp đồng cứng vs Phong cách linh hoạt

Tại sao các hệ thống ngân hàng truyền thống (Core Banking cũ) vẫn bảo thủ trung thành với SOAP?

  • SOAP (Protocol): Dựa trên hợp đồng giao tiếp XML Schema (WSDL) cực kỳ khắt khe. Nếu Client truyền sai kiểu dữ liệu dù chỉ 1 ký tự hoặc thiếu trường, máy chủ SOAP sẽ lập tức từ chối xử lý ngay từ tầng xử lý cú pháp. Đảm bảo tính toàn vẹn tuyệt đối.
  • RESTful (Architecture): Là phong cách kiến trúc mở, linh hoạt hơn. Không ép buộc định dạng (JSON, XML, Multipart). Sử dụng Swagger (OpenAPI) làm bản thiết kế đặc tả kỹ thuật để các phân hệ Front-End bám sát theo.
  • State: SOAP có thể cấu hình Stateful (máy chủ lưu context); RESTful tuyệt đối Stateless (mỗi request độc lập).

2. Security: WS-Security (Tin nhắn) vs TLS/SSL (Đường ống)

So sánh cơ chế bảo mật cốt lõi trong các hệ thống ngân hàng liên kết:

  • SOAP (WS-Security): Mã hóa và ký số trực tiếp vào từng phần tử con (node XML) trong Body tin nhắn. Bảo mật xuyên suốt kể cả khi gói tin đi qua các trạm trung gian (Proxy/ESB) không an toàn trước khi đến đích.
  • REST (TLS/SSL + OAuth2): Chỉ mã hóa toàn bộ "đường ống" truyền nhận HTTPS (Point-to-point). Nếu Proxy trung gian bẻ khóa (Decrypt) được SSL, dữ liệu thô JSON bên trong sẽ lộ hoàn toàn trừ khi nhà phát triển tự viết mã hóa riêng cho Body.

3. Bẫy Hiệu năng bộ nhớ: DOM Parsing (XML) vs Token/Stream Parsing (JSON)

Cơ chế xử lý phân rã chuỗi thô (Parsing) trong RAM quyết định trực tiếp đến giới hạn chịu tải tối đa của hệ thống:

  • DOM Parser (SOAP XML): Máy chủ SOAP bắt buộc phải tải toàn bộ tài liệu XML vào bộ nhớ RAM và xây dựng cấu trúc cây thực thể Tree-structure (Document Object Model) phức tạp. Từng thẻ đóng/mở, thuộc tính và namespace đều biến thành một Node Object riêng biệt chiếm dụng bộ nhớ $\rightarrow$ Tiêu tốn RAM gấp 5 - 10 lần kích thước tệp tin thực tế.
  • Stream/Token Parser (REST JSON): Định dạng JSON phẳng không chứa metadata namespace rườm rà. Bộ parser phân rã chuỗi JSON trực tiếp bằng cơ chế Stream (quét tuần tự từ trái qua phải) và ánh xạ trực tiếp thành các kiểu dữ liệu nguyên bản của ngôn ngữ lập trình $\rightarrow$ Giải phóng RAM ngay lập tức và tăng tốc độ xử lý.
  • Overhead băng thông thực tế: XML sử dụng thẻ đóng mở liên tục khiến dung lượng Payload phình to (Ví dụ: Giao dịch chuyển tiền SOAP tốn khoảng ~1,500 bytes, trong khi REST JSON tương đương chỉ tốn ~350 - 400 bytes) $\rightarrow$ Máy chủ SOAP dễ bị nghẽn CPU và bộ nhớ RAM khi chịu tải cao.
Tác động hạ tầng: Ở hệ thống tải cao (10,000 TPS), CPU của máy chủ SOAP luôn bị đẩy lên 95%-100% để bộ dọn rác Garbage Collection (GC) dọn dẹp hàng triệu Node Object vụn do DOM Parser sinh ra, gây ra lỗi ngắt quãng hệ thống (GC Pause).

4. Chiến lược Test: Săn tìm "Mã lỗi ẩn danh" (The SOAP Fault 200 OK Trap)

Bẫy chết người đối với Tester khi viết kịch bản tự động hóa (Automation Script):

  • Bản chất lỗi: Trong kiến trúc REST, mã lỗi phân tách rõ ràng qua Status Code (4xx, 5xx). Còn với SOAP, máy chủ Web Server đứng ngoài (như IIS, Apache) chỉ đóng vai trò vận chuyển gói tin XML. Nếu gói tin được chuyển đến đích thành công, Web Server vẫn trả về mã mạng HTTP 200 OK mặc dù logic nghiệp vụ bên trong bị lỗi nghiêm trọng.
  • Hậu quả thực tế: Nếu Tester chỉ viết script tự động kiểm tra Status Code (`pm.response.to.have.status(200)`), bộ test-suite sẽ báo trạng thái "Passed" (Màu xanh), bỏ lọt các lỗi chí mạng như: Tài khoản không đủ tiền, sai mã PIN, hoặc giao dịch bị từ chối.
  • Giải pháp viết Script kiểm thử nâng cao trong Postman: Tester bắt buộc phải chuyển đổi chuỗi XML thô trả về thành JSON, định vị sâu vào cấu trúc Node con của SOAP Envelope để rà quét thẻ <soap:Fault> hoặc mã lỗi nội bộ:
Postman Script Săn Lỗi SOAP Fault ẩn danh // Bước 1: Chuyển đổi dữ liệu phản hồi XML sang JSON Object
var xmlJson = xml2Json(pm.response.text());

// Bước 2: Viết Assertion kiểm tra thẻ Fault không được phép xuất hiện
pm.test("Verify hệ thống không trả về thẻ lỗi nghiệp vụ SOAP:Fault", function() {
    var hasFault = xmlJson["soap:Envelope"]["soap:Body"]["soap:Fault"];
    pm.expect(hasFault).to.be.undefined;
});

Anatomy of an API: Cấu trúc cơ bản của một gói tin API

1. Gói tin REQUEST (Client gửi)

Là gói tin thô do trình duyệt/App khách hàng đóng gói truyền đi bao gồm:

  • Method: Động từ hành động xác định (GET, POST, PUT, DELETE).
  • URL Endpoint: Địa chỉ tài nguyên trên mạng (`/v1/transfer`).
  • Request Headers: Metadata chỉ định cấu hình kỹ thuật (Token, Content-Type).
  • Request Body: Toàn bộ dữ liệu nghiệp vụ chính gửi lên Server (JSON/XML).
Raw HTTP Request POST /v1/transfer HTTP/1.1
Host: api.itlearn.vn
Content-Type: application/json
Authorization: Bearer eyJhbGciOi...

{ "account_src": "9991", "amount": 200000 }

2. Gói tin RESPONSE (Server phản hồi)

Là kết quả máy chủ dịch vụ sau khi xử lý nghiệp vụ gửi ngược lại cho khách hàng:

  • Status Code: Mã số định dạng kết quả (200=OK, 4xx=Lỗi Client, 5xx=Vỡ code).
  • Response Headers: Các cấu hình bảo mật, loại tệp tin do Web Server quyết định.
  • Response Body: Trực quan hóa dữ liệu kết quả nghiệp vụ thành công hoặc thất bại.
Raw HTTP Response HTTP/1.1 200 OK
Content-Type: application/json; charset=utf-8
Date: Wed, 15 Jul 2026 11:00:00 GMT

{ "status": "SUCCESS", "tx_id": "FT202611" }

1. HTTP Method - Phân định hành động & Chống trùng lặp dữ liệu

Hệ thống Backend ngân hàng phân định hành động nghiêm ngặt bằng Method để tối ưu hiệu năng và thiết lập cấu hình an toàn:

  • GET (Read): Chỉ dùng để truy vấn dữ liệu (Ví dụ: xem số dư, danh sách thẻ). Cấm truyền dữ liệu trong body để tránh rò rỉ thông tin nhạy cảm trên log server.
  • POST (Create): Dùng khi khởi tạo giao dịch (Ví dụ: tạo lệnh chuyển tiền). Toàn bộ dữ liệu bắt buộc bọc kín trong Body.
  • PUT vs PATCH (Update): PUT dùng ghi đè toàn bộ bản ghi Object (Nếu truyền thiếu trường, dữ liệu cũ sẽ bị gán thành null). PATCH chỉ cập nhật cục bộ một vài thuộc tính (Ví dụ: khóa thẻ tạm thời).
Kịch bản test thực chiến - Idempotency Key: Khi mạng chập chờn, người dùng bấm nút "Chuyển tiền" liên tục. API chuẩn cần cấu hình Header X-Idempotency-Key (chứa mã UUID duy nhất cho mỗi phiên gửi). Nếu trùng khóa trong 30 giây, máy chủ từ chối và trả về lỗi `409 Conflict` để tránh trừ tiền khách hàng hai lần.

2. URL Endpoint - Thiết lập Biên phân vùng & Kịch bản Injection

URL chứa địa nhìn định danh tài nguyên hệ thống. Đây là nơi thường bị khai thác lỗ hổng bảo mật:

  • Kiểm thử giá trị biên độ dài tham số (Boundary Testing): Thử nghiệm truyền mã tài khoản dài quá mức (Ví dụ: 1000 ký tự) xem hệ thống có bộ lọc Validation chặn lại và trả về lỗi `400 Bad Request` hay bị tràn bộ đệm gây sập Server (Lỗi `500`).
  • Kiểm thử bộ lọc ký tự điều hướng (Path Traversal): Chèn các ký tự điều hướng thư mục hệ thống như ../etc/passwd vào tham số URL để kiểm tra lỗi lộ tệp tin nội bộ.
  • SQL / NoSQL Injection: Điền các chuỗi ký tự phá vỡ câu lệnh SQL thô trực tiếp trên tham số URL (Ví dụ: ?cif_id=101%20OR%201=1). API chuẩn bắt buộc phải mã hóa đầu vào (Input Sanitization) để vô hiệu hóa chuỗi mã độc.

3. Header Authorization - Quản trị ma trận xác thực & Lỗi IDOR

Phân biệt rõ ràng hai khái niệm xác thực cốt lõi khi thiết lập kiểm thử bảo mật:

Authentication (Bạn là ai?) - Mã 401

Xác thực danh tính đăng nhập. Token sai, rách, hết hạn $\rightarrow$ Trả lỗi mạng 401 Unauthorized.

Authorization (Bạn có quyền gì?) - Mã 403

Kiểm duyệt quyền hạn. Giao dịch viên cố tình gọi API phê duyệt giải ngân của Giám đốc $\rightarrow$ Trả lỗi 403 Forbidden.

  • Kiểm thử lỗ hổng IDOR (Insecure Direct Object Reference): Lấy Token hợp lệ của khách hàng A để gọi API truy vấn thông tin số dư của khách hàng B (bằng cách thay đổi tham số `account_id` trong request). API thiết kế chuẩn phải kiểm duyệt tính tương thích giữa Token và chủ tài sản, nếu không sẽ bị lộ dữ liệu hàng loạt.

4. Header Content-Type - Bẫy cưỡng ép định dạng dữ liệu

Quy định định dạng truyền nhận dữ liệu giữa Client và Server. Việc kiểm thử sai lệch Header giúp phát hiện lỗi cấu hình của máy chủ:

  • application/json: Chuẩn trao đổi dữ liệu key-value thông dụng nhất của RESTful API.
  • application/x-www-form-urlencoded: Dữ liệu mã hóa dạng chuỗi dính liền `key1=val1&key2=val2`. Thường sử dụng trong tích hợp các cổng thanh toán kết nối link điều hướng (Ví dụ: Napas, VNPAY).
  • multipart/form-data: Sử dụng khi truyền tải file nhị phân kèm dữ liệu văn bản. Ứng dụng thực tế: **Luồng eKYC gửi ảnh căn cước công dân và ảnh chân dung**. Kịch bản test: Tải lên file chứa mã độc có đuôi giả dạng ảnh (Ví dụ: `malware.png.exe`) xem máy chủ có phát hiện và chặn lại hay không.
Kịch bản bẫy "Header đè Body": Client gửi một gói tin có Body là dữ liệu JSON chuẩn nhưng cố tình cấu hình Header `Content-Type: text/plain` hoặc `application/xml`. Nếu Backend không phân tách rõ ràng, nó sẽ bị crash, nhè log thô hoặc trả về lỗi 500 thay vì trả về lỗi chuẩn 415 Unsupported Media Type.

5. Request Body Payload - Phá vỡ logic dữ liệu đầu vào

Nơi lưu trữ nội dung chi tiết của tác vụ. Đây là khu vực tập trung nhiều lỗi logic nghiệp vụ nhất:

  • Kiểm thử giá trị số âm chí mạng (Data Mutation): Điền số tiền giao dịch là số âm `{ "amount": -1000000 }`. Nếu Backend thiếu hàm Validation chặn giá trị $\le 0$, hệ thống khi thực thi phép toán trừ tiền (Số dư - Số tiền chuyển) sẽ biến thành phép cộng (Số dư - (-1000000) = Số dư + 1000000) $\rightarrow$ Tài khoản nguồn tự dưng được cộng thêm tiền khống!
  • Kiểm thử thiếu thuộc tính bắt buộc (Mandatory Field): Thử xóa bỏ thuộc tính bắt buộc `{ "account_destination": null }` xem Backend xử lý lỗi có an toàn không, hay bị lỗi vỡ luồng bộ nhớ `Null Pointer Exception` làm lộ thông tin hệ thống.

6. HTTP Status Code - Định chuẩn mã lỗi Semantic trong kiểm thử thực tế

Status Code phản ánh chính xác kết quả xử lý của tầng mạng và logic nghiệp vụ hệ thống ngân hàng:

NhómMã lỗi sử dụng phổ biếnKịch bản và Cách test thực tế
2xx (Success) `200 OK`, `201 Created` Test luồng Happy Path. `201` dùng khi khởi tạo tài khoản mới thành công, `200` cho truy vấn xem số dư thẻ.
4xx (Client Error) `400 Bad Request`
`409 Conflict`
`422 Unprocessable Entity`		 - `400`: Test gửi chuỗi JSON sai cú pháp (thiếu dấu phẩy, sai ngoặc).
- `409`: Gửi trùng lặp giao dịch liên tiếp để test bộ chặn đúp lệnh.
- `422`: Body đúng cú pháp nhưng lỗi logic (Ví dụ: Số dư không đủ chuyển khoản).
5xx (Server Error) `500 Internal Error`
`504 Gateway Timeout`		 - `500`: Test gửi ký tự đặc biệt phá vỡ câu lệnh SQL, nếu nhè lỗi code thô là Bug.
- `504`: Ép thời gian xử lý cơ sở dữ liệu quá tải để kiểm tra tính năng ngắt kết nối an sau.

7. Response Header - Rà soát Hiệu năng & Cấu hình mạng an toàn

Header phản hồi chứa các chỉ số cấu hình và đo lường hiệu năng tải hệ thống:

  • Verify Content-Type đầu ra: Đảm bảo định dạng phản hồi trả về là `application/json; charset=utf-8` để các thiết bị Mobile hiển thị ký tự tiếng Việt không bị lỗi font chữ.
  • Đo lường thời gian phản hồi (Response Time): API nghiệp vụ lõi (Core Banking) bắt buộc thời gian phản hồi phải dưới **500ms**. Tester cần viết các đoạn script assertion tự động báo lỗi nếu thời gian chờ vượt ngưỡng an toàn để cảnh báo sớm nguy cơ nghẽn hàm xử lý (Bottleneck).

8. Response Body - Kiểm soát Schema & Data Integrity

Nơi hiển thị kết quả cuối cùng trả về cho người dùng. Tester cần kiểm duyệt cực kỳ gắt gao:

  • Kiểm duyệt định dạng Schema (Schema Validation): Đảm bảo các thuộc tính trả về không bị thay đổi định dạng chữ hoa/chữ thường so với tài liệu đặc tả ban đầu (Ví dụ: `transaction_id` bị DEV sửa nhầm thành `transactionId` làm Mobile App không đọc được dữ liệu, hiển thị màn hình trắng).
  • Lỗi bẻ cong kiểu dữ liệu khi Mapping: Số tiền lớn (`10,000,000,000.55` VND) của hệ thống Core Banking cũ khi đi qua API trung gian REST bị DEV ép sai kiểu dữ liệu sang String thô hoặc kiểu số thực float bị làm tròn, bẻ gãy bộ logic tính toán lãi suất chính xác.

Làm chủ JSONPath & Kỹ thuật truy vấn dữ liệu khi Test API

1. Bản chất JSONPath & Các Toán Tử Truy Vấn Core

JSONPath là ngôn ngữ truy vấn dùng để bóc tách, định vị và trích xuất thông tin có cấu trúc từ chuỗi JSON phản hồi:

Toán tử Cú pháp ví dụ Nguyên lý hoạt động và ứng dụng kiểm thử
`$` `$.customer_profile` Trỏ vào phần tử gốc (Root Object) của Response JSON để bắt đầu chuỗi truy vấn.
`.` hoặc `[]` `$.data.id` hoặc `$.data['id']` Truy cập thuộc tính trực tiếp (Child Operator) của một Object.
`..` `$..balance` Deep Scan (Quét đệ quy sâu) tìm tất cả các thuộc tính khớp tên bất kể phân cấp cha con.
`*` `$.accounts[*].id` Wildcard - Trích xuất tất cả các phần tử con hoặc tất cả giá trị trong mảng.
`?()` `$..list[?(@.status == 'ACTIVE')]` Filter Expression - Chỉ lọc ra các Object con thỏa mãn biểu thức logic mong muốn.

2. Quy tắc kiểm tra cấu trúc dữ liệu JSON

  • Kiểm soát nút rỗng (Null/Undefined Safety): Khi phân tích tài liệu API, cần xác định thuộc tính nào là bắt buộc có dữ liệu, thuộc tính nào có thể bị rỗng (`null`) để đối chiếu trên Response thực tế.
  • Xác định chỉ mục phần tử (Index Mapping): Nắm chắc quy tắc phần tử đầu tiên của một mảng luôn có chỉ số index bắt đầu từ số 0 (`[0]`), tránh nhầm lẫn khi đếm dữ liệu trực quan trên màn hình.
  • Đối soát Schema: Kiểm tra xem kiểu dữ liệu trả về thực tế có khớp với đặc tả không (Ví dụ: Số tiền giao dịch phải trả về kiểu Number không có dấu ngoặc kép, nếu trả về dạng String `"500000"` là lệch chuẩn thiết kế).
Phản hồi mẫu phức tạp (Response Body JSON) {
  "cif_customer": "ITLEarn789",
  "status_code": "ACTIVE",
  "accounts": [
    { "acc_no": "1101", "type": "SAVING", "balance": 50000000, "is_locked": false },
    { "acc_no": "1102", "type": "CHECKING", "balance": 1500000, "is_locked": true }
   ]
}

Bản đồ ánh xạ đường dẫn JSONPath thực tế:

  • • Để lấy mã khách hàng: Trỏ đường dẫn $.cif_customer
    ➔ Giá trị trả về: "ITLEarn789"
  • • Để lấy số tài khoản đầu tiên trong mảng: Trỏ đường dẫn $.accounts[0].acc_no
    ➔ Giá trị trả về: "1101"
  • • Để lấy số dư tài khoản thứ hai trong mảng: Trỏ đường dẫn $.accounts[1].balance
    ➔ Giá trị trả về: 1500000
  • • Quét lấy toàn bộ danh sách số tài khoản: Trỏ đường dẫn $.accounts[*].acc_no
    ➔ Kết quả trích xuất: ["1101", "1102"]

Cách nạp các biến động vào cấu trúc URL hoặc Body Payload mà không cần lập trình:

1. Cấu trúc lồng biến:
Bọc tên biến trong cặp ngoặc nhọn kép dạng {{ten_bien}} ở bất kỳ ô nhập liệu nào trên UI của Postman (URL, Headers, Body, Params).

2. Các biến sinh dữ liệu tự động (Dynamic Variables):
Postman hỗ trợ sẵn các biến hệ thống để giả lập dữ liệu thực tế:
- {{$guid}}: Tự động sinh ra một mã UUID ngẫu nhiên không trùng lặp (Ứng dụng cho X-Request-ID).
- {{$timestamp}}: Sinh ra mốc thời gian hiện tại của hệ thống dưới dạng Unix Epoch.
- {{$randomFullName}}: Tự động sinh tên ngẫu nhiên cho kịch bản tạo mới tài khoản.

Nguyên lý kịch bản chuỗi liên kết (Chained Requests) bằng Giao diện

Để chuyển tiếp dữ liệu (Ví dụ: Token, ID giao dịch) từ API trước sang API sau mà không phải sao chép thủ công:

  • Bước 1 - Trích xuất Token: Ở API Login, sau khi nhận được phản hồi thành công, ta thực hiện lưu trữ giá trị Token vào biến môi trường bằng tính năng nạp biến tự động trên giao diện (hoặc thông qua Collection variables).
  • Bước 2 - Lưu trữ biến động: Biến môi trường jwt_banking_token sẽ được ghi nhận trực quan tại mục quản lý Environment.
  • Bước 3 - Nạp tự động ở API tiếp theo: Tại request truy vấn số dư, chỉ cần cấu hình Authorization Header với cú pháp: Bearer {{jwt_banking_token}}. Postman sẽ tự động nạp giá trị token đã lưu mà không cần can thiệp code.

  • 3. Thiếu Assertion bằng mắt trực quan: Chỉ thấy mã trạng thái mạng `200 OK` là bỏ qua bước rà quét logic nghiệp vụ lỗi được mô tả chi tiết hiển thị trong Response Body.

  • 5. Không xóa dữ liệu rác sau khi Test: Tạo tài khoản test liên tục nhưng không dọn dẹp, gây tràn bộ nhớ cơ sở dữ liệu phân hệ thử nghiệm ngân hàng.
